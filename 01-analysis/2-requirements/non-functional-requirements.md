## NFR-001 · P1 · Целевая платформа — Android

**Описание:** Клиентское приложение первой поставки разрабатывается и публикуется **только для Android**.

**Ограничения:**

- Минимальная поддерживаемая версия Android — **API 23 (Android 6.0 Marshmallow)**.
- Целевая версия — последняя стабильная на момент разработки.
- iOS — вне текущего релиза (BR-013).

**Связи:** questions.md §9

---



## NFR-002 · P1 · Mobile-first интерфейс

**Описание:** UI оптимизирован для использования **с телефона в зале скалодрома**: читаемые шрифты, крупные зоны нажатия, вертикальная навигация.

**Обоснование:** «Пользоваться будут в основном с телефона — клиенты прямо в зале».

**Связи:** BR-001

---



## NFR-003 · P1 · Режим работы — только онлайн

**Описание:** Все ключевые операции (просмотр слотов, запись, отмена) требуют **активного сетевого подключения**. Офлайн-кэш записей **не требуется**.

**Связи:** questions.md §9

---



## NFR-004 · P1 · Бэкенд как единственный источник истины

**Описание:** Клиентское приложение и клиентский API **не дублируют** бизнес-логику атомарного бронирования. Состояние слотов, мест, броней и отмен определяется ответами **существующего бэкенда**.

**Обработка ошибок:** Приложение корректно обрабатывает HTTP-статусы, возвращаемые API (см. FR-032).

**Обоснование:** R-004 — граница интеграции; гарантия «0 двойных броней» на стороне сервера.

**Связи:** BR-002, FR-009

---



## NFR-005 · P1 · Каноническая схема данных — контракт API

**Описание:** Все сущности клиентского приложения (слот, бронь, инструктор, зона, прокатный тариф, адрес, прокатный фонд) маппятся на **поля контракта API**. Легаси-данные и миграции **вне скоупа** (R-015).

**Связи:** FR-005, FR-023

## NFR-006 · P1 · Срок поставки — начало сезона (~2 месяца)

**Описание:** MVP должен быть готов к **началу нового сезона** скалодрома (ориентир заказчика — «через пару месяцев»).

**Implication:** приоритизация P1-требований; P3/P4 не блокируют релиз.

**Связи:** BR-003

---



## NFR-007 · P1 · Роль «Клиент» — единственная в поставке

**Описание:** В текущем релизе реализуются сценарии **только для роли Клиент**. Интерфейсы инструктора и администратора **не разрабатываются** (R-028).

**Связи:** BR-014

---



## NFR-008 · P2 · Устойчивость к нестабильной сети в зале

**Описание:** При временной потере связи приложение показывает индикатор загрузки, понятные сообщения об ошибке и возможность **повторить запрос** без потери введённых данных на текущем шаге записи.

**Особенности:** При повторной отправке запроса на бронирование клиентское приложение использует **Idempotency-Key** в заголовках запроса. Бэкенд возвращает статус предыдущей попытки (успех/ошибка) при повторном запросе с тем же ключом.

**Таймаут запроса бронирования:** При ожидании ответа более 5 секунд показывается сообщение: «Сервер временно недоступен. Пожалуйста, повторите попытку» (FR-030).

**Обоснование:** Использование в зале при возможно слабом Wi‑Fi / мобильной сети.

**Связи:** NFR-003

## NFR-009 · P2 · Язык интерфейса — русский

**Описание:** Все пользовательские тексты, сообщения об ошибках и уведомления — на **русском языке**.

**Связи:** brief-climbing.md

## NFR-010 · P2 · Безопасность персональных данных

**Описание:** Номер телефона и профиль клиента передаются по защищённому каналу (HTTPS/TLS). SMS-коды — одноразовые, с ограниченным временем жизни и rate limiting на стороне API.

**Rate limiting:** При превышении лимита попыток получения или ввода SMS-кода приложение показывает сообщение: «Слишком много попыток. Попробуйте через 15 минут» (FR-033).

**Связи:** FR-001, FR-022, FR-033

---



## NFR-011 · P2 · Push- и SMS-интеграция через бэкенд

**Описание:** Регистрация device token / номера для SMS и отправка уведомлений выполняются **серверной частью** (существующий бэкенд + клиентский API). Приложение передаёт необходимые идентификаторы и отображает статус доставки только при наличии в API.

**Push-уведомления:** Доставляются через FCM (Firebase Cloud Messaging) и приходят в любом состоянии приложения: активном, фоновом или убитом системой (FR-034).

**Настройка уведомлений:** Клиент может включить или отключить все push-уведомления разом (FR-029). Критические уведомления (отмена скалодромом) всегда доставляются независимо от настроек.

**Связи:** FR-015, FR-017, FR-018, FR-021, FR-029, FR-034

---



## NFR-012 · P2 · Тестовый стенд для разработки и приёмки

**Описание:** Для интеграции доступен **тестовый стенд** API с репрезентативными данными (слоты, инструкторы, прокат, сценарии отмены).

**Ограничение:** Тестовый стенд должен быть доступен **не позднее чем за 2 недели до начала разработки** клиентского приложения. Задержка готовности стенда является критическим риском для проекта.

**Статус:** ответственность команды бэкенда (questions.md §10).

**Связи:** FR-009

---



## NFR-013 · P3 · Время отклика списка слотов

**Описание:** Загрузка списка слотов на 7 дней при нормальных условиях сети — **не более 3 секунд** (p95) с момента открытия экрана до отображения данных или empty state.

**Измерение:** на эталонном устройстве (Pixel 5 или аналогичное Android-устройство среднего сегмента) при эмуляции сети 4G (скорость ~10 Мбит/с, задержка ~50 мс).

**Связи:** FR-002

---



## NFR-014 · P3 · Доступность empty state и ошибок

**Описание:** Сообщения empty state и ошибок API формулируются **plain language** без технических кодов; код ошибки доступен только в логах / dev-режиме.

**Связи:** FR-004, FR-009

---



## NFR-015 · P2 · Обработка устаревшей версии приложения

**Описание:** При запуске приложение проверяет версию API через эндпоинт `/version`.

**Поведение:**

- Если версия API не поддерживается текущей версией приложения, показывается экран с сообщением: **«Обновите приложение для продолжения работы»** и кнопкой перехода в магазин приложений.
- Критичные операции блокируются до обновления.

**Связи:** NFR-006
