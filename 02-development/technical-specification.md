# Техническое задание на разработку мобильного приложения «Вертикаль»

## 1. Введение

### 1.1. Наименование системы
Мобильное приложение «Вертикаль» для клиентов скалодрома.

### 1.2. Заказчик
Оля, владелица скалодрома «Вертикаль».

### 1.3. Основание для разработки
Бизнес-потребность в автоматизации процесса записи клиентов на групповые тренировки, снижении операционной нагрузки на владельца и повышении качества сервиса.

## 2. Цели и задачи проекта

**Основная цель:** Разработать клиентское мобильное приложение для самостоятельной записи пользователей на тренировки, которое интегрируется с существующей серверной инфраструктурой скалодрома.

**Ключевые задачи:**
- Предоставить клиентам удобный инструмент для просмотра расписания и онлайн-записи.
- Исключить двойные бронирования и ручной труд владельца по управлению записями.
- Внедрить механизмы для снижения количества неявок (напоминания, штрафы).
- Дать обратную связь по работе инструкторов через систему оценок.

## 3. Объект автоматизации

Процесс записи клиентов на групповые тренировки в скалодроме «Вертикаль».

**Автоматизируемые операции:**
- Просмотр актуального расписания.
- Запись на тренировку (с выбором снаряжения).
- Отмена записи.
- Управление профилем клиента.

## 4. Требования к продукту

### 4.1. Функциональные требования
Продукт должен реализовывать полный пользовательский путь, описанный в use-cases и user-stories. Ключевой функционал:

1.  **Авторизация:**
    - Вход и регистрация по номеру телефона и SMS-коду (FR-001, US-001).
    - Автоматический вход по токену (US-012, FR-027).
    - Возможность выхода из приложения.
2.  **Расписание:**
    - Просмотр слотов на 7 дней вперед (FR-002, US-003).
    - Фильтрация расписания по произвольному периоду (до 3 месяцев) (FR-003, US-013).
    - Отображение карточки слота с обязательными атрибутами: время, формат, инструктор с рейтингом, свободные места, цена, адрес (FR-005).
    - `Empty state` при отсутствии слотов в выбранный период (FR-004).
3.  **Запись:**
    - Запись на тренировку только на себя (FR-006).
    - Обязательный выбор снаряжения: «свое» или «прокат» (FR-007).
    - Отображение итоговой стоимости (с прокатом и скидкой лояльности) на экране подтверждения (FR-013, FR-023).
    - Экран успешной записи (FR-008).
    - Push-уведомление о подтверждении записи (FR-021).
4.  **Управление записями:**
    - Просмотр списка предстоящих записей (US-002, FR-012).
    - Отмена записи. Правила отмены: без штрафа (если > 10 мин до начала), со штрафом (если ≤ 10 мин до начала) (US-006, FR-010, FR-011).
    - Просмотр истории записей (US-018, FR-031).
5.  **Оценка инструктора:**
    - Возможность поставить оценку от 1 до 5 после завершенной тренировки (US-009, FR-019).
    - Оценка доступна в течение 24 часов после окончания слота.
6.  **Профиль и лояльность:**
    - Отображение статуса лояльности (счетчик посещений, прогресс до скидки 50% на 10-е посещение) (US-010, FR-020).
    - Отображение статуса штрафных баллов и блокировок (US-017, FR-028).
    - Настройка push-уведомлений (вкл/выкл) (US-019, FR-029).
7.  **Уведомления:**
    - SMS-напоминания за 24 и 2 часа до начала тренировки (US-008, FR-017, FR-018).
    - Push-уведомления: подтверждение записи, приглашение оценить (через 1 час), отмена тренировки скалодромом (US-007, FR-015).

### 4.2. Нефункциональные требования (NFR)

1.  **Платформа:** Android, минимальная поддерживаемая версия — API 23 (Android 6.0) (NFR-001).
2.  **Дизайн:** Mobile-first, адаптированный для использования в зале (крупные элементы, вертикальная навигация) (NFR-002, NFR-009).
3.  **Сеть:** Только онлайн-режим. Офлайн-кэш не предусмотрен (NFR-003).
4.  **Интеграция:** Бэкенд является единым источником истины. Все данные и бизнес-логика (проверка мест) обрабатываются на серверной стороне (NFR-004).
5.  **Производительность:** Загрузка списка слотов не более 3 секунд при нормальных условиях сети (NFR-013).
6.  **Надежность:**
    - Таймаут запроса на бронирование — 5 секунд (FR-030).
    - Повторная отправка запроса на бронирование с `Idempotency-Key` для предотвращения дублей (NFR-008).
    - Корректная обработка HTTP-статусов (400, 401, 403, 404, 409, 422, 429, 500, 503, 504) с понятными пользователю сообщениями (FR-032).
7.  **Безопасность:** Передача данных по HTTPS; сохранение токена сессии в защищенном хранилище (NFR-010).
8.  **Сроки:** Релиз MVP должен быть готов к началу нового сезона (через ~2 месяца) (NFR-006). Интеграционный стенд должен быть доступен за 2 недели до старта разработки (NFR-012).

## 5. Интерфейсы и интеграция

### 5.1. Внешние системы
- **API Бэкенда:** Основная точка взаимодействия. Предоставляет данные о слотах, инструкторах, обрабатывает запросы на бронирование, отмену, регистрацию push-токенов и т.д. API выступает как `black-box`.
- **SMS-шлюз:** Используется для отправки одноразовых кодов при входе и напоминаний о тренировках.
- **Push-сервис (FCM):** Используется для доставки push-уведомлений клиентам.
- **Внешняя система администрирования:** Взаимодействие не требуется. Отмена слота в ней инициирует изменения в API, которые обрабатываются приложением.

### 5.2. Спецификация API (OpenAPI)
Взаимодействие с серверной частью осуществляется строго в соответствии с [OpenAPI спецификацией](./api/openapi.yaml), которая описывает:

- URL-пути к ресурсам (`/auth`, `/schedule`, `/bookings`, `/profile`, `/ratings`).
- Форматы запросов и ответов.
- HTTP-методы и коды ответов.
- Схемы данных (сущности `TrainingSlot`, `Booking`, `Client` и др.).
- Механизм авторизации через Bearer-токен.

## 6. Модель данных (логическая)

Клиентское приложение оперирует следующими основными сущностями (маппинг на поля API):

- `Client`: id, phone, display_name.
- `TrainingSlot`: id, instructor, format, zone, starts_at, ends_at, free_places, base_price, rental_price, rental_available, status, address.
- `Booking`: id, slot, equipment_type (`own`/`rental`), total_price, status (`active`, `cancelled_by_client`, `cancelled_by_club`, `completed`, `no_show`), attended, cancellation_reason.
- `InstructorRating`: score.
- `LoyaltyAccount`: completed_visits_count, visits_to_next_discount.
- `PenaltyAccount`: penalty_points, is_booking_blocked, blocked_until.

## 7. Архитектура приложения

### 7.1. Общие принципы
- **UI:** Single Activity с многоэкранной навигацией (Jetpack Navigation или аналоги).
- **Архитектурный паттерн:** MVVM (Model-View-ViewModel) с Clean Architecture (слои Presentation, Domain, Data).
- **Асинхронность:** Kotlin Coroutines + Flow.
- **DI (Dependency Injection):** Koin или Dagger Hilt.

### 7.2. Структура слоев
1.  **Presentation (UI):**
    - `Activities/Fragments`: Отображение экранов (согласно `screen.md`).
    - `ViewModels`: Содержат бизнес-логику для UI, обрабатывают состояния экрана (Loading, Success, Error).
    - `Adapters`: Для списков (расписание, мои записи).
2.  **Domain (Бизнес-логика):**
    - `Use Cases`: Инкапсулируют отдельные сценарии (например, `GetSlotsUseCase`, `BookSlotUseCase`, `CancelBookingUseCase`).
    - `Repositories Interfaces`: Описывают контракты для получения данных (например, `ScheduleRepository`, `BookingRepository`).
    - `Models`: Классы данных, используемые в приложении.
3.  **Data (Интеграция):**
    - `Repositories Implementation`: Реализуют интерфейсы, получают данные из API.
    - `Remote Data Source (API)`: Содержит реализацию `Retrofit`-клиента для взаимодействия с бэкендом согласно OpenAPI.
    - `Local Data Source (DataStore)`: Для хранения `access_token`, настроек уведомлений и других легковесных данных.

### 7.3. Навигация
Структура навигации и переходы между экранами описаны в разделе «Навигация» файла `screen.md`.

Приложение считается готовым к релизу при выполнении следующих условий:

1.  Весь функционал из раздела 4.1 работает согласно спецификации.
2.  Интеграционные тесты с тестовым стендом API проходят успешно по всем сценариям (успешный, альтернативные и ошибочные потоки).
3.  Все нефункциональные требования (NFR) соблюдены, включая производительность и обработку ошибок.
4.  Приложение протестировано на целевых Android-устройствах.
